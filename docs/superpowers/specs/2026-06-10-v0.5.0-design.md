# portside v0.5.0 — "the floor" — Design Spec

**Date:** 2026-06-10
**Status:** Approved

## Purpose

Remove the blockers that stop strangers from adopting portside, and make the
agent pane agent-agnostic (Claude Code, Codex CLI, anything):

1. **Agent-agnostic launcher** — `work -a <agent>` / `WORK_AGENT` env.
2. **Passphrase & password auth** — interactive prompt in the TUI when
   key/agent auth fails.
3. **ProxyJump** — bastion hosts from ssh config, including hop chains.
4. **File operations** — upload, rename, delete, new folder (both backends).

## 1. Agent-agnostic launcher

- `scripts/work` (bash): `work [-a AGENT] [host|dir]`; default agent =
  `$WORK_AGENT` if set, else `claude`. The right pane runs the agent (local
  mode) or `ssh -t host "bash -lc '<AGENT>'"` (remote mode).
- `scripts/work.ps1`: `-Agent <cmd>` param with the same env fallback.
  The wsl-fallback only applies when the agent is the default `claude`.
- Go: status text in sendto.go becomes "sent to agent pane:" (the mechanism
  already targets whatever runs to the right). README: feature is called
  "send to the agent pane".

## 2. Passphrase & password auth

- `sshconn.Connect(alias, secret string)`. With `secret == ""` behavior is
  unchanged. With a secret, auth adds, in order:
  - encrypted key files decrypted via `ssh.ParsePrivateKeyWithPassphrase`
    (keys that fail to decrypt are skipped),
  - `ssh.Password(secret)`,
  - `ssh.KeyboardInteractive` answering every prompt with the secret.
- `sshconn.IsAuthErr(err)` — true when the ssh handshake failed due to
  authentication ("unable to authenticate" / "no supported methods remain"),
  as opposed to network errors.
- UI flow: `connectCmd(alias, secret)`; `connectResultMsg` gains
  `authFailed bool`. On authFailed, the explorer enters `modePassword`: a
  masked textinput (`EchoMode = EchoPassword`) prompting
  `password for <host>: `; enter retries `connectCmd(alias, typed)`; esc
  cancels. Wrong secret → prompt again (status shows the failure). Secrets
  are kept only in the textinput and the in-flight command, never persisted;
  the input is cleared after each attempt.

## 3. ProxyJump

- `Resolver` reads `ProxyJump` for the alias ("none"/empty → direct).
- Hop chains (`ProxyJump a,b`) connect via a, then b, then the target. Each
  hop may be a config alias or a `[user@]host[:port]` literal; aliases
  resolve recursively through the config (own keys/user/port, and their own
  ProxyJump — depth-capped at 8 to stop cycles).
- Mechanism: dial the first hop normally; for each next hop,
  `prev.Dial("tcp", addr)` + `ssh.NewClientConn` over that net.Conn.
- The same `secret` (if any) is offered to every hop in v1.
- Tested with two chained in-process servers (the test server already
  supports direct-tcpip).

## 4. File operations

`fs.Filesystem` grows four methods, implemented by both backends:

```go
Upload(localSrc, destDir string) error // local file/dir → backend dir
Rename(oldPath, newName string) error  // newName is a bare name, not a path
Remove(path string) error              // recursive
Mkdir(path string) error
```

- Local: copyTree / os.Rename / os.RemoveAll / os.Mkdir.
- SFTP: recursive put (mirror of Download) / client.Rename / recursive
  delete (client.RemoveAll) / client.Mkdir.

Explorer keys + modes:

| Key | Mode | Behavior |
| --- | --- | --- |
| `u` | modeUpload | prompt "upload (local path): "; uploads into the selected dir (or the selected file's parent dir) |
| `m` | modeRename | prompt prefilled with the current name; renames in place |
| `D` | modeDelete | prompt "delete <name>? y/N"; `y` deletes, anything else cancels |
| `n` | modeMkdir | prompt "new folder: "; creates inside the selected dir (or the selected file's parent dir) |

After a successful op the affected directory listing is reloaded in place:
tree nodes get a `parent` pointer; ops reload the parent's children (or the
root listing for top-level entries) so expansion state elsewhere is kept.
Errors go to the status bar as usual. All ops run async as tea.Cmds.

## Out of scope

Per-key passphrase prompts (one shared secret per connect attempt), ssh
config `Match`, agent forwarding to hops, multi-select, trash/undo,
drag-and-drop upload, progress bars for big transfers.

## Testing

- Auth: gliderlabs test server with a `PasswordHandler` (accepts a known
  password) → Connect-with-secret paths, IsAuthErr classification.
- ProxyJump: two in-process servers, config fixture with `ProxyJump`,
  assert the target is reached through the hop (and a 3-hop chain works).
- fs ops: both backends, tempdir + in-process SFTP (same pattern as today).
- UI: Update-level tests for the new modes with the fake filesystem,
  including the password retry loop.
