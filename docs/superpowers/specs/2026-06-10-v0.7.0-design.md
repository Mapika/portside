# portside v0.7.0 — workspace jump, narrow layout — Design Spec

**Date:** 2026-06-10
**Status:** Approved

## 1. Workspace jump (`>`)

One action moves BOTH panes to another directory: the explorer re-roots and
the agent pane restarts there (agents cannot change cwd mid-session, so a
restart is the only correct semantics; documented in README).

- `>` in the tree opens `modeJump` with prompt `workspace: `, prefilled with
  the selected directory's path (selected file → its parent dir; nothing →
  rootPath). Enter confirms, esc cancels.
- On confirm: `loadRootCmd(fsys, dir)` (same as the path bar) plus
  `respawnAgentCmd(dir)`:
  - no `$TMUX` → status "moved the explorer only (agent pane needs tmux)".
  - local backend: `tmux respawn-pane -k -t {right-of} -c <dir> <agent>`
    (argv form, no shell string needed).
  - remote backend (fsys.Name() != "local"): the pane must re-run ssh; one
    shell-command string passed to
    `tmux respawn-pane -k -t {right-of} <cmdstring>` where
    `cmdstring = "ssh -t <shq(host)> -- " + shq("bash -lc "+shq("cd "+shq(dir)+" && exec "+agent))`
    with `shq` = POSIX single-quote escaping (`'` → `'\''`). The command
    construction lives in a pure function returning the argv slice so it is
    unit-testable byte-for-byte.
  - success status: "workspace → <dir>".
- portside learns which agent to respawn via a new `--agent <cmd>` flag
  (default "claude"), threaded App → explorer. The `work` scripts pass their
  agent through (`portside --agent "$AGENT"` / `--agent $Agent`).
- The jump path goes through the same control-character guard as sendto
  (refuse, red status) since it is injected into a shell string in remote
  mode.

## 2. Narrow default layout

- `scripts/work`: right pane size `-l "${WORK_SPLIT:-90%}"` (was 65%).
- `scripts/work.ps1`: `--size` from `$env:WORK_SPLIT` (strip a trailing
  `%`, /100) defaulting to 0.9.
- README documents WORK_SPLIT.

## 3. Width-aware truncation

Every line the UI emits is clipped to the available width — at 10% split the
pane is ~15-25 columns and wrapped lines currently destroy the layout.

- Helpers in a new `internal/ui/text.go`:
  - `truncRight(s string, w int) string` — rune-aware, appends `…` when cut
    (uses lipgloss.Width for display width; strings passed in are PLAIN, not
    yet styled — all call sites truncate before styling).
  - `truncPathLeft(s string, w int) string` — keeps the tail, prepends `…`.
- Applied (all using the explorer's width minus its 1-col left padding):
  - explorer title: host part + left-truncated path;
  - tree rows: indent+marker preserved, name right-truncated to fit;
  - host list rows, the hosts-mode hint line;
  - ports view rows + hints (ports gains a `width` field set from
    WindowSizeMsg like height);
  - App status bar: right-truncated to app width (statusStyle already sets
    Width; add MaxWidth/truncation so it cannot wrap).
- Textinput prompts (path/save-to/etc.) keep bubbles' own behavior (it
  scrolls horizontally already); just ensure the input's Width is set to the
  pane width so it doesn't wrap: set `.Width` on the inputs when
  WindowSizeMsg arrives.

## Out of scope

Persisting the agent's session across jumps (use `claude --continue`),
multi-pane layouts, configurable jump key, truncation of the demo gif.

## Testing

- Pure: truncRight/truncPathLeft tables (ascii, wide runes, exact fit, w<=1);
  respawn argv builder (local + remote quoting torture: dir with space and
  single quote).
- UI: `>` prefill logic (dir selected / file selected / empty), esc, confirm
  produces both a root load and a respawn-or-status cmd (TMUX unset in tests
  → expect the explorer-only status); narrow render: View() at width 18 with
  long names contains `…` and no line exceeding 18 cells (strip ANSI, use
  lipgloss.Width per line).
